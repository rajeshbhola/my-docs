- **Definition**: A class/module should have only one reason to change.

#### âœ… **Dependency Inversion Principle (DIP)**

- **Best Practices**:

  - Define ports in **Hexagonal Architecture**.

### 2. **Separation of Concerns (SoC)**
- **Definition**: Split system into distinct features/modules with minimal overlap.
- **Production Best Practices**:
  - Layered architecture: Controller â†’ Service â†’ Repository.
  - Microservices: Separate auth, billing, inventory.
  - Frontend: Separate UI, state, API logic (e.g., React hooks).

### 3. **Donâ€™t Repeat Yourself (DRY)**
- **Definition**: Every piece of knowledge must have a single, unambiguous representation.
- **Production Caveat**: **Donâ€™t over-apply**â€”clarity > reuse.
- **Best Practices**:
  - Extract shared logic into **libraries** (published to private Maven repo).
  - Use **configuration-as-code** (e.g., shared Helm charts).
- **Anti-Pattern**: Premature abstraction of slightly similar code.

### 4. **Keep It Simple, Stupid (KISS)**
- **Definition**: Simplicity enables reliability.
- **Production Impact**: Fewer bugs, easier onboarding, faster debugging.
- **Best Practices**:
  - Avoid microservices if a modular monolith suffices.
  - Prefer `if-else` over State pattern for 2 states.
  - Delete unused code (YAGNI).

### 5. **You Arenâ€™t Gonna Need It (YAGNI)**
- **Definition**: Implement features only when needed.
- **Production Impact**: Reduces technical debt and attack surface.
- **Best Practices**:
  - Build the simplest thing that works.
  - Defer scalability optimizations until metrics prove necessity.

### 6. **Fail Fast**
- **Definition**: Detect and report errors as early as possible.
- **Production Best Practices**:
  - Validate inputs at boundaries (`@Valid` in Spring).
  - Use **assertions** in critical paths (enabled in staging).
  - Crash on startup if config is invalid (e.g., missing DB URL).

## ğŸ” **III. Resilience & Reliability Principles**

### 7. **Graceful Degradation**
- **Definition**: System continues operating at reduced functionality during failure.
- **Production Best Practices**:
  - Return cached data if DB is down.
  - Disable non-critical features (e.g., recommendations) during high load.
  - Use **circuit breakers** (Resilience4j) with fallbacks.

### 8. **Defense in Depth**
- **Definition**: Apply multiple layers of security/defenses.
- **Production Best Practices**:
  - Input validation + SQL parameterization + RBAC + network policies.
  - Secrets in Vault + encrypted at rest + audit logs.

### 9. **Assume Failure (Chaos Engineering Mindset)**
- **Definition**: Design for inevitable outages (network, disk, third parties).
- **Production Best Practices**:
  - Set timeouts on all external calls.
  - Use **bulkheads** to isolate failures.
  - Run **game days** with controlled failure injection.

### 12. **Principle of Least Privilege**
- **Definition**: Grant minimal permissions needed.
- **Production Best Practices**:
  - Service accounts with scoped DB roles.
  - Kubernetes pods with restricted capabilities.
  - API tokens with short TTLs and narrow scopes.

### 13. **Security by Design**
- **Definition**: Bake security in from day one.
- **Production Best Practices**:
  - SAST/DAST in CI pipeline.
  - Dependency scanning (OWASP DC, Snyk).
  - Regular penetration testing.

### 14. **Test Pyramid**
- **Definition**: Prioritize unit > integration > E2E tests.
- **Production Best Practices**:
  - **70% unit tests** (fast, isolated).
  - **20% integration** (Testcontainers for real DB/Kafka).
  - **10% E2E** (critical user journeys).
- **Anti-Pattern**: Flaky UI tests as primary validation.

### 15. **Contract Testing**
- **Definition**: Ensure services adhere to agreed interfaces.
- **Production Best Practices**:
  - Use **Pact** or **Spring Cloud Contract**.
  - Run in CI before merging.
